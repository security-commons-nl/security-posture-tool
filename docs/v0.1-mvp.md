# v0.1 MVP — data verzamelen en tonen

*Datum: 19 april 2026 · Status: plan, klaar om te bouwen*

## Doel

Een werkende, kleine tool waarmee een security-team **vandaag** begint met het verzamelen en zichtbaar maken van identity-posture-data. Geen framework. Geen abstract plugin-systeem. Geen dashboard-fan. Alleen: data eruit halen, wegschrijven, laten zien. Verder uitbouwen alleen zodra het team bij gebruik tegen iets aanloopt.

Het volledige architectuurplaatje (`architecture.md` + `overview.html`) is een *vision* voor v1.0+. Dat komt als en wanneer de praktijk het vraagt. Deze v0.1 is de empirische startlijn.

## Scope — wat wel, wat niet

**Wel:**
- Eén connector: Microsoft Graph API voor Entra ID
- CSV-upload-endpoint voor handmatige oplevering (kroonjuwelenlijst, Intune-exports, on-prem AD-queries)
- SQLite als opslag — geen Postgres, geen migraties, geen Alembic
- Mini FastAPI + Jinja2-webpagina met tabellen — geen Next.js, geen D3-fan
- Eén `README.md` met "zo draai je het"

**Niet in v0.1:**
- Control-YAML + rule-engine + DSL
- AI-analyse, AI-prioritering, AI-interpreter
- Pending_review workflow
- Severity ≠ Priority
- Evidence SHA-256 hashing
- Pseudonimisering / redactie-laag
- Overlay-mechanisme
- Multi-connector plugin-architectuur
- Prometheus / structured JSON-logging
- Audit-log
- Retention-jobs
- Authenticatie / rolmodel
- Docker-compose (lokaal draaien via `uv run` of `python app.py`)
- D3 DiD-fan
- PDF-export

Alles in deze lijst kan later toegevoegd worden; geen daarvan is nu nodig om waarde te leveren.

## Wat wordt concreet meetbaar

Uit een cyber-hygiene-checklist voor gemeenten (BIO 2.0 / IBD-geïnspireerd, fasering "Week 1") is deze subset met v0.1 direct meetbaar:

| Checklist-ID | Maatregel | Databron in v0.1 |
|---|---|---|
| **3.1** | MFA verplicht op externe toegang + admin-accounts | Graph API — Conditional Access policies + MFA-registration-state |
| **3.4** | LAPS op werkplekken + servers | Graph API — Intune device config policies *of* CSV-upload |
| **3.5** | Inactieve accounts >90 dagen | Graph API — sign-in logs + `signInActivity` |
| **7.2** | Office-macros uit voor internet-bestanden | Graph API — Intune Security Baseline / ASR-rules *of* CSV-upload |
| **1.1** | Kroonjuwelenlijst (max 20, met eigenaar) | CSV-upload |

Daarmee dekt v0.1 vijf van de tien Week 1-items uit de checklist. De overige vijf (sign-in logs naar SIEM, externe scan, gast-WiFi, KPI-rapport, etc.) vragen aparte tooling of zijn eenmalige handelingen.

## Datamodel v0.1 (SQLite)

Plat, drie tabellen. Geen Observation/Finding/Control-scheiding — die komt pas als we meerdere bronnen hebben en aggregatie nodig is.

```sql
CREATE TABLE accounts (
    id TEXT PRIMARY KEY,            -- Entra object-ID of custom
    upn TEXT,
    display_name TEXT,
    is_privileged INTEGER,          -- 0/1
    mfa_registered INTEGER,         -- 0/1
    mfa_methods TEXT,               -- komma-gescheiden
    last_signin_at TEXT,            -- ISO timestamp
    source TEXT,                    -- 'entra' | 'csv'
    collected_at TEXT
);

CREATE TABLE crown_jewels (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    owner TEXT,
    vlan_or_subnet TEXT,
    backup_type TEXT,
    rto TEXT,
    rpo TEXT,
    source TEXT,                    -- 'csv' | 'manual'
    created_at TEXT
);

CREATE TABLE checklist_state (
    checklist_id TEXT PRIMARY KEY,  -- '3.1', '3.4', etc.
    label TEXT,
    measured_value TEXT,            -- bv. "87% dekking" of "12 accounts"
    target TEXT,                    -- "100%" of "0"
    last_measured_at TEXT,
    notes TEXT
);
```

## Werkwijze

1. **Setup** — clone repo, `pip install -r requirements.txt`, vul `.env` met Graph-app-registration credentials (read-only tenant-scope)
2. **Run** — `uvicorn app:app --reload`, browse `http://localhost:8000`
3. **Collect** — klik "Refresh Entra" → backend pullt privileged accounts, MFA-status, sign-in activity en schrijft naar SQLite
4. **Upload** — kroonjuwelenlijst als CSV uploaden, verschijnt in tabel
5. **Review** — per checklist-ID zie je huidige meting, target, en een "doe iets"-knop (bijv. export naar CSV voor opvolging)

## Wat er NIET in staat en bewust is weggelaten

Elke keuze hieronder is een expliciete v0.1-scope-beslissing. Niet vergeten, bewust uitgesteld:

- **Geen gebruikersbeheer.** De tool draait lokaal bij één operator. Als 'ie shared wordt, komt auth.
- **Geen AI.** Geen samenvatting, geen prioritering, geen duiding. Eerst de data. AI voegt pas waarde toe als de dataset groot genoeg is om niet handmatig leesbaar te zijn.
- **Geen Control-framework.** Metingen zijn gehardcoded in `checklist.py`. Als we voor de derde keer dezelfde logica schrijven, refactoren we naar YAML-config.
- **Geen historische trend.** Elke refresh overschrijft de vorige. Als we trend willen, voegen we een `observations`-tabel toe met timestamps.
- **Geen aggregatie over bronnen.** Elke tabel is zijn eigen verhaal. Geen Finding-laag. Komt als we meerdere connectors hebben.

## Acceptatie-criteria voor v0.1

- `README.md` compleet, iemand anders kan 'm lokaal draaien zonder hulp
- Entra-pull werkt tegen een echte tenant
- Minimaal 3 van 5 checklist-items (3.1, 3.5, 1.1 is het makkelijkst) hebben een meetwaarde op de checklist-pagina
- Uploaded CSV voor kroonjuwelen toont in UI
- Totaal <1000 regels Python
